# Architecture

## Goals and layers

Chrono Notes is designed around single data ownership, consistent parsing, replaceable external boundaries, monotonic asynchronous state, and core rules that are testable without Obsidian.

The current plugin ID is `chrono-notes`, and production artifacts live in `dist/chrono-notes/`. The sibling legacy project is read-only input for requirements, verified algorithms, and regression cases. It is not a runtime dependency, must not be modified from this repository, and creates no released-version compatibility obligation; current requirements and the parity checklist define product scope.

```text
app       lifecycle, registration, dependency composition
core      pure models, parsing, date and statistics algorithms
features  use cases, controllers, indexes, immutable query snapshots
adapters  Obsidian Vault/Workspace/DOM and external systems
ui        React calendar views and native settings UI
shared    plugin-local settings, i18n, and small common types
```

Dependencies point inward. Core does not import Obsidian, React, or storage implementations.

## Note data flow

```text
Vault create/modify/rename/delete
              ↓
       ObsidianNoteSource
              ↓
           NoteIndex
              ↓
      parseNoteDocument()
              ↓
          ParsedNote
              ↓
      selectors / React hooks
```

`NoteIndex` is the only source for note existence, content state, frontmatter, previews, tasks, and statistics. Views do not read Vault files or combine an existence lookup with an unrelated statistics cache.

### Data contracts

- `NoteSource` exposes only Markdown path enumeration, complete-text reads, and normalized event subscriptions; it does not leak `TFile` or Vault objects.
- `ParsedNote` combines a path, structured frontmatter, parse failures, preview, tasks, statistics, and one `parseNoteDocument()` result as a recursively frozen value. No derived field may introduce another read path. The YAML root must be a mapping; syntax failures, type failures, and alias-expansion limits remain note-local parse failures rather than masquerading as Vault read failures. Tasks are extracted only from normalized body content while preserving original file line numbers and the legacy Tasks emoji date-marker semantics.
- Range-note parsing is another `ParsedNote` derivative. Only paired boundaries that satisfy the strict grammar produce a frozen range: each must be either a complete zero-padded ISO calendar date, `YYYY-MM-DD`, or a complete ISO datetime beginning with that date and containing at least hours and minutes. Time-only values, reduced-precision years or months, basic-format dates, ordinal dates, and ISO week dates are rejected. Date and datetime values retain their source value, time-presence flag, normalized date key, and stable sort value; day count includes both boundary dates. Same-day ranges are valid. Missing, non-string, invalid, or reversed boundaries produce a note-local `NoteIntervalError`; there is no separate manager cache, and a readable note never becomes a Vault read failure because of malformed range metadata.
- `NoteIndexSnapshot` has a monotonically increasing version and a frozen path record. `parsed`, `error`, and query-derived `missing` are mutually exclusive states, and older snapshots remain unchanged after later events.
- `NoteIndex` subscribes before its initial scan so it cannot miss changes during startup. It deduplicates initial paths, reserves every per-path revision up front, and reads through a bounded worker pool whose default concurrency is 16. `ObsidianNoteSource` uses `Vault.cachedRead()`, but every result still has to pass both path-revision and lifecycle checks. Initial results remain in a separate staging area until every read succeeds or fails explicitly, then merge atomically with already published live entries and notify once. Live events during the scan advance revisions immediately, while ordinary create/modify events enter microtask intent and completion batches without exposing staged initial entries. Delete and rename remain immediate public invalidation barriers: the old path disappears before the final-path read is queued. A lifecycle number rejects unfinished work from before stop or restart; unload also unsubscribes and discards staging, pending intents, and pending commits. If subscription, path enumeration, or internal initialization throws, the start transaction rolls back active/lifecycle state, the source subscription, queues, in-flight reads, staging, per-path revisions, and projections so the same instance can be started again cleanly.

## Concurrency, identity, and cache contract

- Handle create, modify, rename, and delete.
- Deduplicate reads only for the same path, lifecycle, and revision; initial-read concurrency defaults to 16.
- Track a per-path revision and an index lifecycle number; a result may commit only when both remain current.
- Reduce ordinary create/modify events per path into their final `read` intent within one microtask, with at most one read per terminal path. Each reduced group owns a live-batch token. A bounded read slot covers only reading and parsing and is released as soon as computation finishes, while the caller's `refresh()` promise continues waiting for authoritative publication. The first completion schedules a cancellable, injectable macrotask checkpoint: a fully settled fast group publishes once; a group with slow work publishes completed entries at the checkpoint and the remainder in at most one final publication. Stop/restart cancels the checkpoint and invalidates the old token. Delete and rename remain immediate barriers outside that ordinary-batch budget.
- A live read first builds a canonical `ParsedNoteDocument`. If body, raw frontmatter, BOM, line-ending kind, body-start line, and content state are unchanged, it reuses the published entry, skips YAML/range/task/preview/statistics derivation, and neither advances the public version nor notifies. Recovery from a read error and rename always force a complete parse.
- Represent read failures explicitly instead of silently retaining stale data.
- NoteIndex reads a clock and records path-listing, read, document/domain parse, initial/live commit, snapshot-materialization, and listener-notification samples only when its caller explicitly injects a diagnostics sink. Normal plugin composition enables no diagnostics and records neither paths nor content. A diagnostic clock or sink-write failure is fail-open: it disables later diagnostics for that instance without changing snapshot, publication, listener, or refresh-completion semantics. A deterministic 8/16/32 matrix pins algorithmic bounds; the default of 16 still requires real desktop/mobile long-task and heap review.
- Invalidate by settings dependency rather than recomputing the entire vault.
- Normalize extension-changing renames between Markdown and non-Markdown as delete or create events so old paths cannot linger.
- `NoteIndexSnapshot.taskDates` and `NoteIndexSnapshot.intervals` are incremental immutable sub-snapshots owned exclusively by NoteIndex. Initial and live publication both pass the batch's final per-path contributions once to `replaceBatch()`: task projection copies and sorts each affected bucket once, while interval projection sorts once after every contribution in the batch has been replaced. A sub-snapshot changes revision and identity only for a real domain change; a no-op batch retains its existing buckets, array, and sub-snapshot identity. Neither projection is a separate manager or a second owner of Vault facts.

## Commands, queries, and UI

Immutable indexes serve queries. Use cases own creation, opening, templates, task rescheduling, and range-note writes. React reducers keep only interaction state; Vault-derived data uses external-store snapshots. Settings remain native Obsidian UI.

Shared `selectCalendarDay()` takes one zone-free `LocalDate`, one `NoteIndexSnapshot`, and one `IcsEventIndexSnapshot` and returns a frozen `CalendarDay`. On top of the canonical periodic-note state, preview, and `ParsedNote.statistics`, that model contains calendar extensions, regional holidays and workday metadata, the date's complete ICS-event array, and an optional heatmap metric. The month query first asks core for the four to six complete weeks intersecting the target month, then returns a week-nested `weeks[]` model. Each week contains its start, week-year/number, one `weeklyNote`, seven `CalendarDay`-derived month cells, and a mode-dependent range layout; the root records both NoteIndex and ICS snapshot versions explicitly. Ordinary Month mode builds the real range layout. Heatmap Month mode retains the same recursively frozen shape with empty range layouts and never reads the interval projection. The week query reuses the same `CalendarDay` selector for its detailed seven-day model instead of introducing a month-only ICS projection, view-local existence cache, or duplicate task summary. Complete ICS arrays remain in the query and accessible model; presentation alone chooses how many summaries are visible at each container density. React only renders these models, and the shared indicator applies `todoAnnotationMode` without reading Vault or files from a cell. Month separates semantic selection from its date roving tab stop: a visible selected day wins, then visible today, then the first in-month or otherwise enabled date. Hidden outside-month heatmap cells do not participate, so week/month selection and a selection from another query still leave exactly one visible date tab stop. ArrowRight on a week number moves day-granularity selection and focus to that week's first enabled date. Hover/focus previews render through a portal after a delay and use a pure anchor-and-viewport placement calculation that repositions on scroll or resize. Disabling previews schedules no new tooltip and immediately clears pending timers, the active portal, and its ARIA relationship. Year preview is enabled only in heatmap mode while the setting is on; disabling it, switching to summary, or changing year cancels timers and the active portal, and a render gate prevents a stale tooltip flash.

Beyond the seven shared `CalendarDay` values, the week query adds canonical boundaries, weekly-note state, dated tasks, and week range segments from the same NoteIndex snapshot. It reads exactly seven target buckets from `taskDates.byDate` instead of scanning all notes. NoteIndex updates due/scheduled/start contributions incrementally per committed file, merges date kinds for the same task and day, and keeps the result stably ordered by date, path, and source line. Overdue state is still calculated at query time and compares only an incomplete task's valid due date with dynamic today; scheduled and start dates never create overdue state by themselves. React orders each detailed day as status/work-rest marker, weekday, date, calendar extensions, holidays, and ICS. The seven-day overview is a labelled selection group of native buttons, each using `aria-pressed` for the selected date; because the component has neither ARIA-grid rows nor roving focus, it must not masquerade as `grid/gridcell`. The range Gantt renders after the seven day buttons as an independent normal-flow sibling section. It is not inside the day cells, and its interactive bars remain independent buttons outside the day buttons, preventing both nested interactive controls and a duplicate range list. No-range state renders no explanatory paragraph and keeps only a compact create action when creation is configured; a zero-task section keeps its heading and count without a second empty-state paragraph. Only the occurrence in its own due-date group can be rescheduled. Desktop drag and a native seven-day select in the task row provide equivalent pointer, keyboard, and touch paths; both call the same task command, wait for Vault events, and never write Vault from the component.

`TaskCommands` isolates completion toggles and due-date moves. Pure core logic reparses the target line from the latest source using path, line, and the complete expected task identity, then changes only the checkbox or an existing due marker while preserving LF, CRLF, CR, and mixed endings. A missing line or changed identity is rejected. The Obsidian file port uses `Vault.process` so latest-content validation and replacement occur in one atomic operation; the command never mutates NoteIndex directly. A workspace port opens Markdown and positions and scrolls the editor to the original source line.

Heatmap grading is a pure core rule. Word, link, and tag counts use a positive integer setting step; task completion uses a fixed 25% step; every dimension is capped at levels 0–4. The shared day selector can derive frozen values and levels from the same `ParsedNote.statistics`, with missing, read-error, and unconfigured paths explicitly represented as zero. React only switches render modes. Month heatmap mode hides outside-month content and interaction while retaining dimension, raw value, and level in ARIA and previews, and it does not mount `MonthIntervalStrip`; ordinary Month mode retains the complete range Gantt. Month and year queries consume the same persisted `statisticDisplayDimension` and step. When either independent toggle is enabled, the toolbar reuses one selector-and-legend structure; the Month legend uses the month-cell accent scale and the Year legend uses the annual green scale. Week view instantiates no heatmap tools.

Year view uses separate overview and heatmap selectors, each accepting exactly one `NoteIndexSnapshot`. Overview creates only four quarters and twelve periodic-note summaries; it does not allocate 365 or 366 daily heatmap cells. Heatmap mode alone builds the full year's daily metrics, previews, and error states. Month and quarter paths, note states, and `ParsedNote.statistics` come from one snapshot and remain frozen in the selected result; React performs no follow-up reads per quarter or day. The quarter-segmented summary reuses the shared task indicator while task ARIA remains independent of its visible mode, and the heatmap reuses the core four-dimension grading rule. Every monthly heatmap grid exposes seven actual `row` owners for its daily `gridcell` elements and uses explicit grid row/column placement to retain the column-flow geometry. Arrow keys therefore follow the visual axes: Left/Right move seven days and Up/Down move one. A rendered selected day in the displayed year is the sole roving tab stop; otherwise January 1 is the only temporary fallback, so a selection from another displayed year cannot leave the grid without a tab stop. The UI initially renders two quarters, keeps stable-height placeholders for the rest, and uses `IntersectionObserver` with adjacent-quarter overscan to reveal later segments. Keyboard movement across a quarter boundary expands that same window explicitly. The This month action normalizes selection to the current year's `{ month, day: 1 }` month anchor; YearView mounts the target quarter and handles its scroll once so later observer updates do not steal scroll position.

Calendar extensions implement the core `CalendarOverlayProvider` contract: a `LocalDate` and display locale produce a structural date label, optional event, transition kind, and complete accessible text. Features own an explicit frozen static registry in product order; settings and the shared day selector use provider IDs and never discover modules at runtime. `CalendarDay` maps two ordered slots into recursively frozen `calendarOverlays`, independently of `holidayRegions`, so month and week reuse the same result. The Chinese-lunar provider calculates lunar dates, leap months, solar terms, and traditional festivals without letting an event overwrite the structural date. The Ganzhi provider shows a day pillar normally and the new month pillar on solar-term transitions while retaining the complete year/month/day pillars. Core's caller-owned `LunarDateContext` wraps one solar-to-lunar conversion shared by both providers' from-context entry points. Direct registry selection of both overlays also creates that context once per date, without a module-global cache. A separate core adapter contains `lunar-typescript` global-language switching around one synchronous call and restores it in `finally`. The legacy module-lifetime LRU/Manager cache with mixed responsibilities is not migrated. The new `CalendarDecorationCache` retains only frozen decoration results that are independent of notes and ICS, split across bounded LRUs for final combinations, individual overlay providers, individual holiday-region providers, and date-only lunar contexts. A final key includes date, canonical locale, and both ordered slot lists; provider keys contain date, locale, and one provider ID; and a context key contains only the date. Changing a slot combination can therefore reuse provider objects and the same-date base conversion while preserving selection order and identity. All four layers independently use the same default 2,048-entry bound, one mounted calendar view owns them, and unmount clears them together; `size` reports only the final-combination layer. Underlying providers remain pure and directly testable without the cache.

An independent Mainland China holiday core provider reads statutory rest-day and adjusted-workday data from `lunar-typescript`, returning frozen regional holidays, workday metadata, and explicit data coverage. An ordinary workday and a year outside the dataset cannot share the same semantic state; Traditional Chinese names are mapped at the provider boundary. The shared day selector attaches this metadata only when `holidayRegions` contains `cn`, independently of `calendarOverlays`. React renders only the `CalendarDay` rest/work marker, accessible name, and preview metadata and never calls the holiday library directly.

An independent Singapore provider reads bundled annual public-holiday snapshots. The 2026 data is checked against the [Singapore Ministry of Manpower](https://www.mom.gov.sg/newsroom/press-releases/2025/0616-public-holidays-for-2026), and 2027 against the [same ministry's 2027 announcement](https://www.mom.gov.sg/newsroom/press-releases/2026/0618-public-holidays-for-2027); both include the applicable Monday public holidays following Sunday holidays. It returns frozen trilingual values and explicit coverage without producing Mainland-style adjusted-workday markers. A feature-layer static holiday registry unifies both providers and maps three ordered `holidayRegions` slots into frozen names, workday metadata, and one semantic `work / rest / holiday` marker. Adjusted workdays outrank slot order; ordinary markers follow slot order. React maps the one semantic marker through the active translator to Chinese `班 / 休 / 公假` or English `Work / Off / PH`, without hard-coded providers, regional-language coupling, or runtime module scanning.

External ICS data does not enter NoteIndex or masquerade as Vault notes. Core uses `ical.js` for RFC 5545 components, folded lines, and escaped text, then Luxon normalizes UTC, IANA TZID, and floating times into an explicit display zone. All-day ends remain exclusive, and a default end is supplied only when both `DTEND` and `DURATION` are absent. An explicit end that is invalid, not later than the start, or mismatched between all-day and timed types makes that `VEVENT` invalid, as does an invalid, non-positive, or all-day-incompatible duration; providing both end forms is invalid as well. These events are counted as invalid and skipped rather than silently repaired. Cancellation is excluded before recurrence accounting, so a cancelled event carrying RRULE/RDATE does not count as recurring; other recurrence rules are explicitly counted and skipped. A separate `IcsEventIndex` builds a frozen date index from a read-only source port, retains per-source success or error state, and reuses one source read only within the same request revision. A same-source refresh across revisions starts a genuinely fresh read immediately, then checks the revision again after reading and before parsing, so stale content cannot overwrite newer settings, disablement, or re-enablement. The index separates state `version` from event `contentVersion`: refreshing, `refreshedAt`, and diagnostic changes may advance state without changing content identity. A ready result compares sorted occurrences semantically by date, reuses every unchanged bucket, and reuses the complete `eventsByDate` object without advancing `contentVersion` when all events match. `stop()` idempotently releases subscriptions and the complete event mapping while invalidating late refreshes; unload does not publish an empty state to listeners. The shared `CalendarDay` selector references the complete, stably sorted event array for its date from exactly one index snapshot; month and week depend on content identity and do not split or rebuild the event set independently. Presentation alone folds it: a regular month cell shows at most three summaries and narrow/coarse layouts may hide them all; wide, medium, and narrow week cells show at most three, one, and zero summaries respectively and calculate `+N` from the visible count. Folding never changes the accessible name, which retains every title, source, and continuation state. The Obsidian adapter only reads Vault files, desktop absolute paths, and desktop-vault-relative paths. A Vault-relative source must prefer adapter reads when the file cache is not ready, while desktop filesystem modules are loaded only inside the local-path branch through Obsidian's CommonJS `require`; browser dynamic `import()` must not survive in the bundle. Views and settings never read local files directly.

The Obsidian `ItemView` owns only the React-root lifecycle and plugin use-case bridge. At any moment, each mounted calendar owns one request-bound, view-scoped `CalendarQueryStore`. Its month, week, or year request is copied and frozen when constructed, and render-time `getSnapshot()` never replaces the request used by the committed subscription. A request change creates a candidate store that replaces and disposes the old one only after React commits, so speculative rendering cannot cause tearing or missed notifications. The store subscribes to NoteIndex and the independent IcsEventIndex but retains only that request's latest input/output. Dependency checks for ordinary Month and Week requests include visible path entries, seven dated-task buckets, the relevant interval sub-snapshot, and ICS date-bucket references. A heatmap Month request explicitly excludes interval dependencies, so an interval-only update does not wake that view; unaffected day/week/month/quarter objects remain structurally shared. With no subscribers, `getSnapshot()` returns its existing query snapshot without recollecting dependencies when both NoteIndex and ICS snapshot object identities are unchanged; a source notification with those same identities also skips evaluation. `useSyncExternalStore` wakes React only when the active query identity actually changes. Publishers iterate the listener set as it existed when notification began, so a callback's subscription mutations affect only the next publication. A view switch computes the newly active model, and closing the view disposes the store rather than accumulating results by historical revision. Settings effects are computed from the last successfully persisted normalized snapshot to the next one and routed independently to calendar, Navbar, interval list, or ICS. The interval-list revision advances only for a real interval-list dependency change; persistence-only changes such as first-use and confirmation preferences do not refresh every surface.

Runtime i18n lives in shared and depends on neither Obsidian nor React. The Simplified Chinese catalog is the authoritative source of type-safe message keys; English and Traditional Chinese are complete `Record` values that must cover the same keys at compile time, and runtime does not accept partial catalogs that rely on missing-message fallback. `auto` distinguishes Simplified and Traditional Chinese from system script or region subtags and falls back other unknown languages to English. The translator owns parameter interpolation and count forms through `Intl.PluralRules`, and all three catalogs must preserve the same placeholder set per message, while dates, months, and numbers remain with `Intl`. Components must not add scattered locale branches or private message tables.

The Note Navbar feature selector accepts only a file path, the `intervals` sub-snapshot, and periodic/range settings. It produces a frozen context only for a complete reverse-path match against an enabled periodic rule. Previous, next, and higher targets reuse canonical period anchors. Higher navigation skips not only disabled or empty-pattern types but also any configured target whose generated path cannot complete a `format → parse` round trip, so it never presents an unusable higher action. Weekly/monthly related ranges filter the same incremental interval projection by period overlap. An Obsidian manager keeps a mount map keyed by `WorkspaceLeaf` and uses each periodic `MarkdownView`'s public `contentEl` as a stable anchor, mounting one independent React root immediately before it while responding to active-leaf, layout-change, file-open, Vault rename, and targeted settings changes. The parent mounted class and container are removed together on remount, leaf close, setting disablement, or a file leaving its complete periodic path; sidebar focus leaves open Markdown mounts intact. The Navbar is a `flex: 0 0 auto` toolbar outside editor scrolling, followed by a shrinkable `view-content` flex item. Only phone floating-navigation and auto-full-screen modes reuse the host's `--view-header-top-offset + --view-header-height` to place it below the fixed native header, zero the host's outer top reservation, and restore ordinary Markdown inset plus an opaque top mask so safe-area/header space is not counted twice. Ordinary phones, tablets, and desktop never receive another global `env(safe-area-inset-top)`. The component's `useSyncExternalStore` returns only the stable interval sub-snapshot, so an ordinary non-range note update does not rerender Navbar; it never reads Vault. Presentation uses a symmetric three-column CSS grid to center previous/date/next plus the optional higher-period command as one navigation group while retaining open-calendar in the trailing column; narrow containers hide only higher-action text without creating another DOM or command path.

Date selection uses a pure feature model to validate `LocalDate`, month movement, and the four legacy text formats while reusing the core four-to-six-week boundary grid. The React mini calendar owns only its displayed month, focus, and submission lock. Its title row centers previous month, a fixed-width year/month trigger, next month, and Today as one group; its date grid declares six fixed-height data-row tracks but renders only boundary weeks supplied by the model, stabilizing modal height without inventing wholly unrelated weeks. An Obsidian modal returns the selected date to a main-view navigation request or periodic-note command. Global date navigation still moves the main view to the target month, while the Navbar uses the same modal to open the current note type. Main Week view bypasses that modal and uses mutually exclusive week-year/week popovers driven by the pure week model; selection updates only current date, month context, and day-granularity selection without changing view type. Command-palette mini-calendar and direct-jump entries retain the original path and neither parse note content nor read Vault.

The date context menu first builds a frozen feature-layer action model from configuration and NoteIndex existence state. An Obsidian adapter maps those actions to `Menu`, clipboard, and `Notice` APIs. The adapter explicitly uses Obsidian's HTML-rendered menu mode (`setUseNativeMenu(false)`) so the menu, groups, and icons remain visible in current desktop releases. React passes only the date and snapshot state; open/create still invokes periodic-note commands, and the menu never reads or writes Vault directly.

Periodic note commands isolate side effects behind `PeriodicNoteFilePort`, `PeriodicNoteTemplatePort`, and `PeriodicNoteWorkspacePort`. Each command instance coordinates in-progress creation by canonical target path: concurrent requests create and populate that path only once, waiters report `created: false`, and every request retains its own open target and runs its requested cascade independently after the shared main file completes. A failure propagates to waiters and clears the coordination entry for retry; a main-note template failure still deletes the file created by that attempt. Cascade failures roll back per note and continue with later periods. A view-opening failure after a successful write never deletes user content. Commands do not mutate `NoteIndex`; only Vault events advance query state.

`IntervalNoteCommands` owns reversed-date normalization, the minimum two-day creation rule, deterministic paths, existing-file opening, confirmation, and minimal Markdown writes. Its command instance coordinates in-progress creation by that deterministic path, so concurrent requests write once, waiters report `created: false`, and each still opens with its own target; a failure removes the coordination entry for retry. Its file port owns only existence checks, parent folders, and atomic creation, while the workspace uses the shared open port. A view-opening failure after a successful range-note write likewise never deletes user content. The date-context adapter only launches the input modal and delegates the final operation to this use case; it never touches Vault directly.

Range queries consume the stably sorted frozen array from exactly one `NoteIndexSnapshot.intervals` sub-snapshot, filter by the range folder, custom folder, or entire Vault, and never scan ordinary non-range notes. NoteIndex incrementally removes and adds per-path interval contributions while preserving the start epoch, end epoch, title, and path total order. Folder matching uses complete path segments. Explicit timestamp offsets can reverse epoch order relative to local `dateKey` order, so window filtering and lane allocation must not terminate early on a local date; each ordinary Month week's structural-sharing dependency covers the complete array prefix actually processed through the last item relevant to that week. A pure feature layout greedily assigns the lowest available lane within the visible date window using inclusive day overlap, while a deterministic path hash maps each item to one of eight categorical color indexes. The ordinary Month selector lays out the complete core-generated four-to-six-week boundary window once so cross-week segments retain lane and color, then derives clipped columns, continuation flags, lane limits, and hidden titles. The heatmap Month selector does not access `NoteIndexSnapshot.intervals`, filter ranges, or allocate lanes and returns a frozen empty range layout for every week. The Week query continues to use the ordinary range-layout function. Ordinary Month and Week React views share one renderer and open paths through the workspace port. The legacy `IntervalNoteManager`, view-private date index, and duplicated strip algorithms are not recreated.

The range-note list consumes that frozen result and applies a pure feature selector for case-insensitive title/path search, current-month or current-year overlap filtering, and stable ascending or descending ordering. Its native modal's `useSyncExternalStore` returns only the stable `intervals` sub-snapshot and separately subscribes to the range-scope settings revision. Ordinary non-range note updates and persistence-only settings changes therefore do not wake the list. Opening and creation still use the shared workspace and command ports; no manual-refresh cache is introduced.

The template port explicitly selects either the built-in engine or Templater from settings. The built-in engine supports only the documented `date/time/title` placeholders. The Templater adapter runs after target-file creation and injects a frozen `tp_calendar` target-period context. An unavailable or failing Templater raises an explicit error and never silently falls back to the built-in engine.

Settings use a versioned schema. Current schema v15 represents periodic-note and range-note creation confirmation independently as `confirmPeriodicNoteCreation` and `confirmIntervalNoteCreation`; it removes the shared `confirmBeforeCreate` field without translating the old value into either new toggle. Schema v14 uses `showNoteIndicators: boolean`, removes the runtime `indicatorPosition` field and validator, and deliberately does not translate an old position into the new toggle. Schemas v10 through v13 introduce quarter naming, font mode/factor, task annotation, and Properties date interception. `calendarOverlays` normalization accepts only static-registry IDs, keeps first occurrence order, deduplicates, and caps the list at two; the settings UI presents two ordered dropdown slots and excludes the provider already used by the other slot. `holidayRegions` likewise retains only known static-provider IDs, first occurrence order, and at most three entries. Native settings expose neutral Region 1/2/3 ordered slots, exclude duplicates, and compact after clearing an earlier slot. Invalid enums fall back, the factor is floored and clamped to 0–20, and boolean fields accept only actual booleans. Loading reads raw data, migrates by version, validates and normalizes it, then produces runtime settings; final normalization drops removed fields. Missing, invalid, or pre-v1 versions are treated as v1; future versions are copied without being downgraded by an older migrator. The unpublished My Calendar plugin used a different plugin ID, so its flat settings are not an automatic migration input.

Periodic-path settings persist only the existing `pattern` and `templatePath`; there is no derived folder field. A pure UI helper first recognizes common unquoted Moment tokens, then calls core `formatPeriodicNotePath()` and `parsePeriodicNotePath()` to produce frozen `empty / invalid / valid` previews with a specific reason. Apart from the explicit Obsidian-syntax correction, settings feedback remains identical to actual index recognition. Native Obsidian input suggestions enumerate only Vault folder metadata or paths from `getMarkdownFiles()`. Folder selection writes a canonically Luxon-escaped literal into the same pattern while preserving the filename pattern; template selection writes the complete Markdown path. This UI adapter reads no file content, parses no note, and creates no note-state source beside NoteIndex.

Opening daily notes from date properties consists of a pure ISO-date input parser and an Obsidian capture-phase adapter. The adapter suppresses the original event only when the setting is enabled, daily notes are configured, a primary click hits the open icon of a date-typed property inside `.metadata-properties`, and the input value is valid. It sends the date and default/new-tab target to the unified periodic-note command. Wiki/Markdown links in Properties, body links, non-date properties, non-primary clicks, unconfigured daily notes, and invalid dates all pass through. The adapter neither reads the Vault nor creates another note-state source.

Periodic dates use a time- and zone-free `LocalDate` domain value. Core owns strict zero-padded key parsing and formatting, equality and ordering, rejection of impossible civil dates, and a UTC formatting bridge without host-zone drift. Daily, weekly, monthly, quarterly, and yearly dates first normalize to one anchor, and path formatting, reverse path recognition, navigation, and template context reuse those canonical helpers. `isSamePeriod()` compares only the canonical anchors produced by `getPeriodAnchor()` and receives the week-start rule explicitly. A dedicated `calendar-week` core uses ISO Monday as the week-year/number reference; Sunday-first display extends that boundary one day earlier rather than reading the selected Sunday's own ISO identity. It owns week identity, dynamic 52/53-week enumeration, cross-year ranges, week selection, and week-year selection. Selection preserves the offset from displayed week start and clamps W53 to W52 when the target year has only 52 weeks. Luxon stays inside core date algorithms and pattern compatibility; it is not a UI or adapter data type. System-clock access is isolated in a shared local-date clock that reads local year/month/day fields rather than a UTC date and calculates the next local midnight. The calendar root uses `useLocalToday()` to resynchronize at midnight and again when a backgrounded document becomes visible; presentation then derives the unified `is-current-period` immediately from that value and core `isSamePeriod()`. Today styling and today-dependent overdue queries update without resetting the user's selection or navigation context. Current state never enters the month/week/year queries, NoteIndex or ICS snapshots, any cache, or settings; the period picker and mini calendar retain their independent state implementations.

Settings have a schema version and follow read, migrate, validate, normalize, and targeted-effect stages.

Month renders every week-number button as an independent cell with the same `5px` rounding on all four corners as its date cells, replacing the old continuous-rail shape whose leading corners were rounded and trailing corners square. In Year summary, the month/quarter status overlay is fixed at `top: 4px`, creating about `2px` of visible separation from the label without changing the entry height or the label's geometric center.

Source CSS is bundled into one scoped `styles.css`. A shared `CalendarNoteIndicator` serves month, week, and year consumers from normalized `showNoteIndicators` and the same `none / color / hole` task treatment. Disabling visibility unmounts it; `none` falls back to the note-state symbol; unfinished `color` uses a warning-filled track, unfinished `hole` uses a warning hollow track, and complete progress uses the success color. The component calculates one clamped completion fraction, writes horizontal fill width, and retains completed/total counts in data and parent ARIA without fraction text inside the cell. A month day heading keeps a two-level structure: a top accessory grid with state/progress on the left and the one regional marker on the right, followed by a full-width centered date. Week-number controls and year-summary month/quarter controls instead place status in an absolute top overlay while keeping the label centered against the complete button; an unconfigured period, heatmap mode, or hidden indicators mounts neither the overlay nor an empty status row. Remaining day content follows one vertical track in this order: calendar extensions, region-grouped holiday names, then ICS. A day cell keeps one compact shared base minimum. Holiday names and ICS use content-driven auto rows, reserve no dedicated empty block when absent, and let only their week grow uniformly when visible content needs more space. One extension uses a centered column; two use equal columns at regular widths and stack below an approximately `460px` calendar width without changing type size. The month grid uses an explicit spacer column between the week index and dates. Each week block spans the outer month grid, establishes a same-column subgrid, and defines a date-content track plus a range track. In ordinary Month mode, the week number, spacer, and all seven date-button backgrounds and borders span both tracks, while content inside each date button occupies only the first track; the range Gantt participates in normal flow on the second track, after ICS content and before the date buttons' bottom borders. Its container is a sibling of the day-button row, and its interactive bars remain independent buttons outside the day buttons; nesting them is forbidden. When there are no ranges, the Gantt is not mounted and the second track collapses to `0`. One or more lanes grow only their own week block, leaving every other week at its independent height. Heatmap Month mode mounts no `MonthIntervalStrip`, range button, or `+N` row; date content spans the week block's available tracks and remains centered. The implementation must not simulate range space with an absolute overlay, estimated day-cell bottom padding, or a parallel height function. Every visible ordinary-Month lane remains its own non-overlapping interactive button row, with a thin visual line centered inside by a pseudo-element; the optional `+N` row, inter-row gaps, and bottom breathing space contribute naturally to the week-block height. Fine pointers use a compact lane rhythm, while coarse pointers retain a larger operable row without adding overlapping transparent hit layers. Month range colors adjust toward theme foreground.

The ordinary-Month strip inherits the seven date columns through `subgrid`, including `6px` column gaps normally and `4px` at `≤360px`; the Week Gantt uses the same `6px / 4px` values as its seven-day overview. Range bars therefore retain exact date-column alignment at every supported density. The Month renderer explicitly materializes all `visibleLaneCount` row tracks before the optional overflow track, so a clipped week that contains only a lower continuing lane cannot collapse that segment into the first visual row.

Main-calendar date and period controls use one `is-current-period` class, with current, selected, and keyboard focus as mutually orthogonal presentation states. Current on month days, week days, week numbers, year-view month/quarter entries, and year-heatmap days is a layout-neutral and pointer-transparent neutral `1px` inner border drawn flush with the cell's inner edge by a pseudo-element. The default border color mixes `text-normal` with transparency; Month/Year heat levels 3–4 instead use a higher-contrast border based on `text-on-accent`. Current changes no control background, text color, weight, or decoration. A separate layout-neutral and pointer-transparent layer draws selected as an accent `2px` frame genuinely outside the cell boundary; keyboard focus is another independent outermost frame beyond selection. All three states may coexist without overriding one another's semantics. Grid gaps reserve real space for the outer frames, which must not be simulated with an inset outline or inner offset. Plugin-scoped CSS gives every native button and select inside Calendar, Navbar, and the range list a visible `:focus-visible` outer frame; date cells retain their own outermost pseudo-element focus ring instead, avoiding both host-theme focus loss and a double ring. On an ordinary Month day, these pointer-transparent frame pseudo-elements paint in dedicated layers above the range strip, while the day button itself creates no isolating stacking context and is not raised above the strip; the frames therefore remain continuous without stealing range-button hit testing. Temporary month range preview/endpoints and week task drop targets take visual priority and suppress the current inner border; selected and focus frames remain independent. Exact-day controls emit `aria-current="date"`; week, month, and quarter period controls emit `aria-current="true"`; the appropriate `aria-selected` or `aria-pressed` continues to express selection independently. The main month grid, seven-day week overview, and year summary use a `6px` cell gap at regular widths and tighten to `4px` at `≤360px`; ordinary month-day content uses a `3px` gap, detailed week-day content uses a `2px` gap plus `1px` top breathing space, and the year-heatmap grid uses `4px` normally and `3px` at `≤360px`. Heatmap month-day main content spans the week block's available tracks and centers the date horizontally and vertically with zero internal gap, so the absence of a range strip leaves no offset or empty track. This main-grid rule is not reused directly by the period picker or mini calendar. Within its own scope, the period picker represents current through `.is-current` accent text plus `aria-current` and renders no dot; `.is-selected` draws only a `2px` inset outline plus `aria-pressed`, without setting selected text color or background; `:focus-visible` uses a separate outer outline, and current may coexist with selected. Current month is constrained by both the current year and browsed year, while current week is constrained by both the current ISO week-year and browsed week-year. The mini calendar retains its independent implementation. Ordinary outside-month cells fade only their inner content and static surface, so interactive states do not inherit that opacity.

A pure UI formatter derives quarter names from normalized `quarterNameMode` and the translator. Numeric mode uses the runtime language catalog, while Chinese mode maps quarters 1–4 to `春/夏/秋/冬`. Year-summary labels, year-heatmap quarter headings, and the month picker share this formatter; note paths, period anchors, and Note Navbar labels are not appearance consumers.

The calendar root owns four font variables for header, normal, small, and micro text. `follow-obsidian` references Obsidian UI tokens directly. `follow-widget` changes only at three container-query buckets: at or below 360 px, the middle range, and at or above 720 px, avoiding continuous viewport scaling. `immutable` uses a pure helper to map the normalized 0–20 factor to fixed pixel variables. Week view additionally renders one DOM through three container-density tiers: wide shows the full weekday and up to three ICS summaries, medium shows the short weekday and one ICS summary, and narrow uses the narrowest weekday/date while folding detailed content. Status/work-rest, selection, today, and complete ARIA semantics survive every tier. The week picker establishes its own inline-size container: it defaults to three columns and changes only to two below `360px`, without shrinking type or removing ranges; `ResizeObserver` supplies the same column threshold to keyboard movement. It creates one locale formatter and one frozen 52/53-item presentation list per open popup, then uses bounded vertical scrolling rather than adding anything to NoteIndex or calendar-query caches.

Every selector remains scoped by a `chrono-notes-` control class or plugin root; no unscoped global `.is-current-period` or `.is-selected` rule is allowed. The legacy project contributes verified requirements, pure algorithms, data, and regression cases—not its distributed Manager caches or duplicated file reads.

## Performance and readiness boundaries

The `lunar-typescript` language scope wraps synchronous work only. A target equal to the current language must not trigger the library's full message update; if nested work mutates global language or throws, `finally` still restores the entry language on demand. The scope never crosses `await`.

While settings are visible, one disposable shared path catalog retains path-sorted folder and Markdown-file metadata. Vault create/delete/rename events only mark it dirty for lazy rebuild; modify does not. Empty queries take the candidate limit directly. Fuzzy queries scan every candidate to preserve Obsidian ranking but use a bounded top-K structure for the actual suggestions instead of sorting every match. Hiding settings unregisters Vault listeners and clears the catalog.

Normal calendar queries still require the complete initial NoteIndex because an empty snapshot's `missing` value cannot represent “not indexed yet.” Any progressive-startup change must first introduce separate stable, subscribable readiness and prevent normal selectors and creation commands from interpreting unknown entries as absent before ready. Merely removing the plugin startup `await` is not a safe optimization.
